## Project

`hypeGRL` is a Python library for Graph Representation Learning in hyperbolic (and Euclidean) spaces. Its key contributions are:
1. A unified encoder-decoder framework across multiple embedding methods and geometries
2. Principled handling of **partially observed graphs** (unknown edges) via joint optimization тАФ rather than zero-imputing missing entries, unknown edge weights are treated as free variables optimized jointly with embeddings

The Python package is `hypegrl` (all lowercase), installed from this directory.

## Architecture

The library is organized around the encoder-decoder formalism. Every method encodes a graph into a low-dimensional manifold and defines a decoder that reconstructs pairwise similarity. Unknown edges are handled uniformly across all gradient-based methods.

### Core abstractions

**`hypegrl/embedders/base.py` тАФ `HyperbolicEmbedder` (ABC)**
All embedders inherit from this. The mandatory interface is:
- `fit(G, unknown_edges, X_init)` тАФ encode the graph into embeddings. `X_init` is either `(N, d)` coordinates in the method's native input chart (ball for most, hyperboloid for Lorentz) **or a `Representation`** (any chart тАФ re-charted into the optimisation chart via `from_polar`).
- `embeddings()` тЖТ `(N, d)` **Poincar├й-ball** coordinates (the interop chart тАФ plotters/downstream; saturates past `rтЙИ12`)
- `embeddings_representation()` тЖТ the fitted `Representation` (exact geometry, chart-agnostic via its `to_polar`/`to_ball`/`to_hyperboloid`); concrete base method backed by `self._rep`, `None` before `fit`
- `structural_similarity(G)` тЖТ `(N, N)` тАФ the target the encoder tries to match (e.g., adjacency matrix, forest matrix)
- `decode(X)` тЖТ `(N, N)` тАФ the decoder output (e.g., connection probabilities). `X` is coordinates **or a `Representation`** (distance-based decoders use the exact `rep.dist()`; the curvature-aware Hydra decoders coerce a rep to ball, so `decode(rep) == decode(embeddings())`).

There is **no `distance()` method** тАФ the loss is not part of the ABC. Each embedder expresses its loss as an inline `loss_fn(rep, A)` closure (typically a `_loss_from_dist(rep.dist(), тАж)` helper) passed to the optimiser; see below.

Capability flags (`is_gradient_based`, `is_generative`, `supports_update`, `supports_node_update`) let callers query what a method supports without try/except.

**The `Representation` abstraction (`hypegrl/representations/`)** тАФ the swappable *chart* an embedding is stored and optimised in, orthogonal to the method. Canonical hub is **polar `(r, v)`**; subclasses `PolarRepresentation` (default for neutral methods), `BallRepresentation`, `HyperboloidRepresentation`. Interface: `from_polar`/`to_polar`/`parameters`/`dist` per subclass; the base provides `from_ball`/`from_hyperboloid`/`to_ball`/`to_hyperboloid` via the exact converters (`manifolds/conversions.py`). Every gradient embedder takes a `representation=` constructor arg selecting the chart: **polar** for the neutral methods (`PoincareMaps`, `HyperMap`, `DMercator`, `HydraPlus`), the **eponymous** chart for `PoincareEmbeddings` (`"ball"`) and `LorentzEmbeddings` (`"hyperboloid"`). Polar is numerically robust at the large radii real graphs need (see the representation refactor design notes). **Known caveat**: the polar optimiser step is *not* the true Riemannian gradient of H^{D+1} (product metric + product retraction, not the warped-metric geodesic) тАФ same minima, different path. The exact warped-polar `geoopt.Manifold` now exists as the opt-in `representation="exact_polar"` (`ExactPolarRepresentation` / `WarpedPolarHyperboloid`), but it does **not** supersede this caveat: it fixes the geometry without uniformly improving the optimisation тАФ see the exact-polar design note below. **Practical consequence тАФ the polar chart is scaled differently from ball/hyperboloid under a shared `lr`, so the default `lr_X=1e-2` (tuned in the ball/Riemannian era) can *under-drive* it.** Because the radial coordinate is a plain `Euclidean` parameter (not a point moving under the warped hyperbolic metric), a fixed `lr` produces a different effective geometric step than `RiemannianAdam` on the ball. Two faces of the same mismatch, both traced this session: (1) *under-convergence at small radius* тАФ `PoincareMaps` on `balanced_tree(2,3)` at `lr_X=1e-2` plateaus (loss тЙИ 3.9 at 500 steps) while ball reaches its basin fast; bumping to `lr_XтЙИ3e-2` (or more steps) lets polar reach a *deeper* basin (тЙИ 0.19) that ball never finds тАФ averaged over seeds polar is actually better, seed 0 is just a slow-converging worst case (this drove the `streaming` tutorial to `n_steps=500`); (2) *overshoot at large radius* тАФ HyperMap on deep trees, where the angular gradient scales `~sinh┬▓r` (the [HyperMap lr] note). We deliberately keep `lr_X=1e-2` as the neutral default (it is safe and polar is the robust general chart), and tune per-tutorial where a short step budget exposes the gap; a per-chart default `lr` (or the planned warped-polar manifold) would remove the need. Not a quality regression тАФ a step-size scaling, characterised, not yet defaulted-around.

**`hypegrl/inference/` тАФ `riemannian_optimize()` (fixed target) and `joint_optimize()` (unknown edges)**
Both consume a `Representation` (never bare coordinates). Gradient embedders delegate to one of them from `fit()`:
- `riemannian_optimize(representation, s_A, loss_fn, тАж)` тАФ precomputes the fixed structural target `s(A)` once; optimises `representation.parameters()` (which may span several manifolds, e.g. polar's `Euclidean` radius + `Sphere` direction) by `RiemannianAdam`. Loss is `loss_fn(rep, s_A)`.
- `joint_optimize(G, representation, loss_fn, unknown_edges, тАж)` тАФ rebuilds the adjacency `A` from the current `a_Omega` estimate **each step** (sigmoid-reparametrised, initialised via `imputation.compute_a_omega_init()`), optimising `representation.parameters()` + `a_Omega` jointly. Loss is `loss_fn(rep, A)`.

The loss receives the **representation** (not a bare distance matrix), so the decoder pulls what it needs тАФ `rep.dist()` today (a future `rep.inner()` for RDPG-style decoders) тАФ keeping `L(s(A), decode(X))` chart-agnostic. Each embedder does `if not unknown_edges: riemannian_optimize else: joint_optimize`. Adding a new gradient-based embedder means writing a `loss_fn(rep, A)` (via `rep.dist()`), building the chosen `Representation` from the warm start (`representations.build_representation`), and calling the two optimisers тАФ then reading back `self._X = rep.to_ball()` and `self._rep = rep`.

### Manifold helpers

**`hypegrl/manifolds/poincare.py`** тАФ shared `POINCARE_BALL = geoopt.PoincareBall()` instance (curvature `c=1`), plus coordinate conversion utilities (both NumPy and Torch/autograd versions):
- Poincar├й тЖФ H┬▓ polar (2D)
- Poincar├й тЖФ hyperspherical (d-dim)
- Poincar├й тЖФ Lorentz/hyperboloid

**`hypegrl/manifolds/lorentz.py`** тАФ `StableLorentz` (a `geoopt.Lorentz` subclass) plus the shared `LORENTZ = StableLorentz()` instance (curvature `k=1`). `StableLorentz` clamps each point's spatial norm `тАЦx'тАЦ` to `max_norm` inside `projx` and the retraction (`expmap`/`retr`), so every `RiemannianAdam` step (via `retr_transp тЖТ retr`) stays bounded. This prevents the hyperboloid `NaN` failure тАФ one high-lr `expmap` step scales coords by `cosh(тАЦvтАЦ_L)` and overflows `x_0` тАФ mirroring the reference impl's `set_dim0` renorm (which uses `1e2`). The clamp is overflow-safe (factors out the max coordinate, so an inf-squared-norm point projects onto the boundary in-direction, not to the origin) and keeps the previous point on a non-finite step. **`max_norm` is not just a safety valve тАФ it also regularises spread**: a sweep showed `тЙ▓30` binds below the natural radius and hurts, `тЙ│1e6` lets the fit run away and degrades quality (trees collapse at high lr), with a usable window `~1e2тАУ1e4`; **default `1e3`** (Poincar├й radius `тЙИ0.999`) dominated the reference's `1e2` (equal on trees that never reach it, `+0.8` AUC on denser graphs that do). It's a constructor arg here and on `LorentzEmbeddingsEmbedder` (which builds its *own* instance rather than mutating the shared `LORENTZ`) because the optimum is graph-dependent. Defined on the manifold (not in an embedder) so any future hyperboloid method can reuse it; the D-Mercator Lorentz attempt was abandoned for exactly this overflow (it refines on the Poincar├й ball instead тАФ see that design note).

### Parameter estimation (`hypegrl/inference/parameters.py`)

**Scope / where estimators go.** This module is the single home for estimating the **scalar hyperparameters of the latent-geometry model** from an observed graph тАФ the quantities a method needs *before* it embeds (e.g. HyperMap / E-PSO's `(m, L, ╬│, T, ╬╢)`). It is deliberately method-agnostic and imports no embedders, so any method can reuse it.

Decision rule for what belongs here:
- **Belongs here:** anything that maps a graph (or its degree sequence / clustering) to a scalar model parameter тАФ `╬│` (power-law exponent), `k_min` (power-law cutoff), and, when added, `T` (temperature, via clustering matching) and the `m`/`L` average-degree heuristics.
- **Does *not* belong here:** estimating the *unknown adjacency entries* `a_╬й` (that is `imputation.py`), solving for embeddings (`joint_optimizer.py` / `riemannian_optimizer.py`), or anything specific to one embedder's warm-start pipeline (e.g. the greedy init in `_hypermap_init.py`, the ╬║/╬▓ inference baked into `_dmercator_init.py`).

Current contents:
- `estimate_gamma(G, k_min=None, fallback_gamma=DEFAULT_GAMMA)` тАФ discrete power-law MLE (ClausetтАУShaliziтАУNewman, arXiv:0706.1062, ┬з3.2). `k_min=None` (default) selects the cutoff automatically by KS minimisation; an explicit integer uses that cutoff directly. When no cutoff can be selected (`choose_kmin_ks` returns `None`: < 3 distinct degrees, or no cutoff with a well-supported tail), it warns and returns `fallback_gamma` (`DEFAULT_GAMMA = 2.5`, a neutral scale-free value; chosen over fitting because the `k_min=1` MLE can return an invalid `╬│ < 2`). Also importable as `hypegrl.embedders.hypermap.estimate_gamma`.
- `choose_kmin_ks(degrees, min_tail=DEFAULT_MIN_TAIL)` тАФ CSN ┬з3.3 KS-minimising choice of the cutoff. Candidates are the distinct degrees **with the largest dropped** (following the `powerlaw` package's `find_xmin`): the max-degree cutoff has a single-value tail that any distribution fits perfectly (KS=0), so scoring it spuriously wins (this was giving trees a meaningless `╬│тЙИ6.5`). Candidates whose tail retains fewer than `min_tail` degrees are also skipped (`DEFAULT_MIN_TAIL = 50`, CSN ┬з3.2's "n тЙ│ 50 for reliable estimates" rule of thumb; used as a hard candidate filter is our choice). Without it, pure KS-min drifts into the sparse upper tail of a bell-shaped sequence and reports a spuriously steep `╬│` off a handful of points. **The floor bounds tail size, not `╬│` тАФ the exponent is left uncapped, because HyperMap/E-PSO admits any `╬│ тЙе 2` (╬▓ = 1/(╬│тИТ1) тИИ (0,1]); we deliberately do *not* adopt `powerlaw`'s ╬▒ тЙд 3 normalizability cap, which would reject legitimate `╬│ > 3`.** Returns `{k_min, gamma, ks, n_tail}` (large `ks` тЗТ not really a power law, e.g. a tree), or `None` when there are < 3 distinct degrees or no candidate reaches `min_tail` (pass `min_tail=1` to disable). Consequence: graphs too small for a 50-degree tail (e.g. karate, N=34) now return `None` тЖТ `estimate_gamma` falls back.
- `power_law_gof(degrees, n_bootstrap=1000, seed=None)` тАФ CSN ┬з4 semiparametric-bootstrap goodness-of-fit for the power-law hypothesis; **user-called, never automatic**. Returns `{p_value, plausible, D, k_min, gamma, n_tail, n_bootstrap}` with `plausible = p_value >= 0.1`, or `None` when the data can't be fit (`choose_kmin_ks` тЖТ `None`). Small `p` **rejects** the power law; large `p` only means it's *plausible*, not correct/best. Robustly rejects body-dominated non-power-laws (trees, `pтЙИ0` across seeds) and passes true power laws (BA, zipf). **Known method limitation (not a bug):** a bell-shaped sequence (Poisson/Erd┼СsтАУR├йnyi) is *not* robustly rejected тАФ its sparse upper tail is weakly power-law-compatible, so the single-distribution GOF can call it plausible. Discriminating power-law from such an alternative is the job of a likelihood-ratio test (e.g. `powerlaw`'s `distribution_compare`), which we don't implement.

The E-PSO temperature `T` is **not** estimated here тАФ see the `HyperMapEmbedder.estimate_temperature` design note below. Planned for this module: the `m` / `L` average-degree heuristics (both pure graph тЖТ scalar).

## Key design notes

- **Node ordering тАФ the uniform `nodes()` accessor**: every embedder exposes `emb.nodes()` (a method on the base `HyperbolicEmbedder`, backed by `self._nodes`, set in `fit`) returning the node labels in `embeddings()` row order тАФ `None` before fit. Reordering methods report their order (`HyperMapEmbedder` sorts by degree during greedy init; `DMercatorEmbedder` has its own order); non-reordering methods (`PoincareMapsEmbedder`, `Hydra`, `HydraPlus`, `PoincareEmbeddings`) return `list(G.nodes())`. This replaced the old per-embedder heterogeneity (HyperMap's `nodes_sorted` property, DMercator's `nodes` property). Consequence (a real silent footgun): `plot_poincare_graph`/`plot_polar` and any adjacency you build to pair with the rows must use that order, else every node is drawn at the wrong point and the plot looks like noise тАФ pass `nodes=emb.nodes()` and the plotters align internally (`nodes=None` assumes `G.nodes()` order). Documented in the base `nodes()`, `HyperMapEmbedder.embeddings()`, and both plotter docstrings.
- **HyperMap temperature estimation is post-embedding, not in `parameters.py`**: neither the HyperMap paper nor the official C++ estimates `T` (the C++ takes it as a required `-t` input). The E-PSO paper (ref [5], Papadopoulos, Psomas & Krioukov, IEEE/ACM ToN 23, 198 (2015), ┬зV) *does*: its global connection probability (Eq. 10) is Fermi-Dirac in the hyperbolic distance, `p(x)=1/(1+e^{(╬╢/2T)(xтИТR)})`, so its steepness is exactly `╬╢/2T`. `HyperMapEmbedder.estimate_temperature(X=None, A=None)` reads `T` off a **fixed embedding** by a one-parameter logistic MLE of the edge indicator on `u_ij = x_ij тИТ R_{max(i,j)}` (`p=╧Г(тИТs┬╖u)`, `s=╬╢/2T`, `T=╬╢/2s`) тАФ the max-likelihood form of the paper's binned "tail slope" read-off (our reformulation, not verbatim). Because it needs an embedding (chicken-and-egg with `fit`), it lives on the embedder, **not** in `inference/parameters.py`. `T` is an *output* (it characterises the network's average clustering and calibrates the connection-probability decoder), not a knob to improve the coordinates: the paper reports (its Fig. 5) that embedding *quality* is not significantly affected by the input `T`, so reading `T` off a single embedding made with any reasonable input already gives a good estimate. For a better, self-consistent value, set `self.T` to the result and refit a couple of times. (Distance-ranked link prediction needs no `T` at all; `T` only calibrates the probability values.) Verified to recover a known `T` to <1% on synthetic Fermi-Dirac graphs (`tests/test_embedders.py::test_hypermap_estimate_temperature_recovers_known_slope`). We deliberately chose this over clustering-matching (the D-Mercator-style route via `infer_kappa_and_beta`), which would fit `parameters.py` but needs an E-PSO generator we don't have.
- **Connectivity constraint**: `update()` raises `ValueError` if removing edges/nodes would disconnect the graph тАФ a disconnected Laplacian produces a block-diagonal forest matrix that changes the problem character.
- **`update()` is currently warm-started refit**: both `PoincareMapsEmbedder` and `HyperMapEmbedder` declare `supports_update=True` but implement `update()` as a warm-started call to `fit()`. True incremental updates via Woodbury are planned but not yet implemented.
- **All tensors use `float64`**: the optimization is numerically sensitive; `torch.float64` throughout.
- **Hydra embedders are non-gradient**: `HydraEmbedder` and `HydraPlusEmbedder` use closed-form spectral decomposition. They fall back to zero-imputation with a warning when `unknown_edges` is provided, since they cannot route through `joint_optimize`.
- **The exact warped-polar manifold (`representation="exact_polar"`) тАФ correct geometry, not a better optimiser; opt-in, not the default.** `manifolds/polar.py::WarpedPolarHyperboloid` is a real `geoopt.Manifold` implementing H^{D+1}'s true geodesic-polar geometry: the warped metric `g = dr┬▓ + sinh┬▓(r)┬╖g_{S^D}` (`inner`), the natural gradient `GтБ╗┬╣ = diag(1, sinhтБ╗┬▓r┬╖I)` (`egrad2rgrad` тАФ the `/sinh┬▓r` the product metric drops), and the exact geodesic as `retr`/`expmap`. `ExactPolarRepresentation` packs `[r, v]` into **one** `ManifoldParameter` тАФ the warp couples `r` and `v`, so they cannot live on a `Euclidean`├Ч`Sphere` pair as `PolarRepresentation`'s do (that pair *is* the product-metric approximation). The radius is stored raw (no softplus): the manifold keeps `r тЙе 0` itself and flips `v` to the antipode for a geodesic through the origin. **Numerics**: the exponential map is evaluated through `tanh r` and a log-space radius increment `╬Фr = log(C + тАЦnтАЦ) тИТ log1p(tanh rтВА)`, so no `e^r`/`cosh r` ever forms тАФ verified exact against an independently-computed hyperboloid geodesic to ~1e-16, and still exact at `r = 80` where ball and hyperboloid are long dead. Two flagged engineering choices (ours, not from any reference): a `max_step` cap on the geodesic step length (bounds `cosh s`; with the correct metric the natural step is `тЙИ lr`, so it never binds тАФ a valve, not a knob) and an `╬╡_warp` floor on `sinh┬▓r` in `egrad2rgrad` (polar coordinates are genuinely singular at the origin, so a root node at `r=0` gets a bounded near-zero angular gradient instead of `NaN`). Transport is projection transport, matching `geoopt.Sphere`'s own choice тАФ momentum only, `O(lr)` distortion per step; exact parallel transport (angular rescale `sinh rтВА/sinh rтВБ` + in-plane rotation) is a noted refinement. **The empirical finding is the important part: the exact gradient is *not* uniformly better.** With `lr` now a true radius-independent geodesic step length, `PoincareMaps`/`n_steps=500` gives тАФ on `balanced_tree(2,3)` тАФ the *deepest basin of any chart* (1.155 at `lr=0.1`, vs polar's best 1.807 and ball's 1.542), but on `balanced_tree(2,5)` it **loses to product-metric polar at every `lr`** (best 35.8 vs 20.5), with a narrower stable `lr` window (degrades by `lr=0.3`; ordinary over-stepping, not a numerical failure тАФ points stay finite). **Why:** hyperbolic volume grows like `e^r`, so separating two leaves at radius `r` by `╬Ф╬╕` costs arc length `тЙИ sinh(r)┬╖╬Ф╬╕`; a *correct* step of length `lr` therefore makes glacial angular progress exactly where deep trees need it, while the product metric's "wrong" scaling moves angular *coordinates* at `~lr` regardless of radius тАФ an accelerating preconditioner. Natural gradient тЙа fastest descent, sharpened by `H^n`'s exponential volume growth. Related: `component_inner` returns one **scalar** metric norm per point, so Adam gives radial and angular a single shared adaptive scale (and the `O(sinhтБ╗┬▓r)` angular term is dominated by the radial one); a measured per-block variant (what `ProductManifold` does, and what the split-parameter polar chart gets for free) recovers most of the deep-tree gap (71тЖТ31 at `lr=0.01`) but forfeits the small-tree win and still loses to polar тАФ we keep the scalar form as the coordinate-invariant choice that makes the step length exactly `lr`. So `polar` stays the default; `exact_polar` earns its place as a principled optimiser with an interpretable `lr`, the best small-graph basin, and a reference implementation of the true geometry to measure the approximation against. Whether it wins on the real (single-cell / airport) graphs is open and wants the `tune_inits.py` harness, not another tree. Full derivation: `hypeGRL_cicada/exact_polar_manifold_derivation.md`; tests: `tests/test_representations_exact_polar.py`.
- **Representations are curvature-free by design; only the hyperboloid holds its own `_manifold`.** The chart (`Representation`) is orthogonal to the *metric scale* (curvature), and they are handled in different layers. **Curvature is a global scalar, not a chart property:** H of curvature тИТc is isometric to the c=1 space under `d тЖж d┬╖тИЪc`, so it is fully **absorbable into the loss** тАФ scaling `rep.dist()` by a distance-temperature `╧Д = тИЪc` *is* changing the curvature. Hydra+ already exploits exactly this (`s_A = D┬╖тИЪk` optimised on the c=1 chart тЙб embedding at curvature k; see `_stress_loss_from_dist`); Fermi-Dirac folds the scale into `╬▓`/`R╠В`/`╬╢`; the ranking losses set it implicitly via radial spread (and could gain an explicit `╧Д`). So **every rep fixes curvature `k=1`** (`POINCARE_BALL`, `StableLorentz`, `polar_distances`) and no `Representation` carries curvature; a uniform, optionally-learnable loss-layer `╧Д` is the planned way to make it first-class for all methods (see the TODO.md entry). Adding curvature to a rep would be redundant with the scale the loss already carries and would conflate "how points are stored" with "how big the space is." **Consequence for `self._manifold`:** a rep holds its own manifold instance only when that manifold carries *per-instance state* тАФ only `StableLorentz` does (`max_norm`, a graph-dependent numerical clamp that also drives RiemannianAdam's retraction), so `HyperboloidRepresentation` builds a fresh instance per fit and stores it. `PoincareBall(c=1)` / `Sphere` / `Euclidean` are fixed and parameterless, so `BallRepresentation` / `PolarRepresentation` reference the shared module-level singletons directly (and polar is not even a single geoopt manifold тАФ it is `S^D ├Ч тДЭтВК` with a functional distance, so there is no `self._manifold` to hold).
- **D-Mercator = faithful original init + hyperbolic refinement**: `embedders/_dmercator_init.py` (`dmercator_init`) runs the full original D-Mercator pipeline тАФ joint ╬║/╬▓ inference (clustering matching), S^D-corrected Laplacian Eigenmaps, likelihood-maximisation angular refinement, and final ╬║ readjustment тАФ to produce the warm start, mirroring how `_hypermap_init.py` provides HyperMap's greedy init. `DMercatorEmbedder.fit()` then builds a `Representation` from the native warm start `(v_i, r_i)` (chart chosen by `representation=`, **default polar** тАФ see the representation refactor notes) and refines it with Riemannian Adam, minimising a Fermi-Dirac NLL on the exact hyperbolic distances `p_ij = 1/(1+e^{(╬▓/2)(d_H тИТ R╠В)})`. `embeddings()` returns the ball image; the exact geometry is `embeddings_representation()` / `native_coordinates()`. ╬▓ and R╠В are fixed from the init; the refinement moves both angular and radial coordinates, so the hidden degree ╬║ (recovered from the final radius) is learned jointly with the angle. The init is binary-only (edge weights ignored) and treats ╬▓ as inferred unless fixed in the constructor. Verified that `D=1` reduces to Mercator (`R = N/2╧А`).
- **╬▓ inference is bidirectional with a `╬▓ = D` floor**: clustering matching (`infer_kappa_and_beta`) raises ╬▓ when the model under-clusters and lowers it toward the floor `D + 0.01` when it over-clusters, clamping (with a warning) when the target clustering is unreachable. Low-clustering graphs тАФ trees, where `c╠Д_emp = 0` тАФ drive ╬▓ to the floor. This mirrors the reference C++ `infer_parameters(dim)`/`infer_parameters()`; an earlier one-directional "grow ╬▓ only" bracket left ╬▓ pinned at its random init on such graphs, corrupting ╬╝/R╠В/radial scale.
- **Stage-2 init: LE for all D by default; optional Mercator ordering-init for D=1**: the reference C++ uses LE only for DтЙе2 and the classic Mercator ordering + expected-angular-gap re-spacing for D=1. The `d1_init` constructor argument `DMercatorEmbedder(..., d1_init=...)` (and the `dmercator_init(..., d1_init=...)` function param) default to `"le"` (the paper's generalisation, used for all D); pass `d1_init="mercator"` to use `mercator_ordering_init` (the C++ D=1 init) instead. The flag is ignored (with a warning) for `d > 2`. On leaf-heavy graphs (trees) `"mercator"` spaces degree-one nodes deterministically around their hub while `"le"` reinserts each leaf at a random angle (MLE never moves leaves): measured on `balanced_tree(2,4)`/`(2,5)` over 8 seeds it gives a ~6тАУ10├Ч larger minimum inter-leaf angle and comparable-to-better reconstruction (AUC 0.83 vs 0.77 on `(2,4)`, ~tied on `(2,5)`). Default stays `"le"`.
- **Why D-Mercator defaults to the polar chart (the numerics behind `representation=`)**: polar, ball, and hyperboloid are all exact charts of H^{D+1} (each `Representation`'s `dist()` is the exact hyperbolic distance), so the choice is purely numerical тАФ and it matters because low-degree leaves belong at large radius `r тЙИ R╠В` (~16тАУ20 even on tiny graphs, up to ~40 on larger ones). At those radii: the **hyperboloid** timelike coordinate `cosh(r) тЙИ 1e5тАУ1e7` overflows off the manifold during optimisation (fatal in the D=1/Mercator, leaf-heavy regime тАФ NaN on most seeds of a Barab├бsiтАУAlbert tree); the **ball** stays bounded in (тИТ1, 1) but *saturates* тАФ `tanh(r/2) тЖТ 1` collapses every leaf past `r тЙИ 12` onto the boundary, so their pairwise distances are wrong; the **polar** chart `(r, v)` keeps `r` a plain moderate number (no `e^r`, no saturation) and is exact at all radii. Hence **`representation="polar"` is the default** (`"ball"`/`"hyperboloid"` remain selectable, with those failure modes). An earlier Lorentz/`geoopt.Lorentz` refinement was abandoned for the overflow, which motivated the whole representation refactor; `tests/test_embedders.py::test_dmercator_robust_on_leaf_heavy_graph_d1` guards against a leaf-heavy NaN regression. (Full three-chart comparison: `experiments/representation_comparison.py`.)
- **Poincar├й Embeddings has two interchangeable losses, default = ranking**: `PoincareEmbeddingsEmbedder(loss=...)` selects the encoder-decoder instantiation. `"ranking"` (default, the original Nickel & Kiela 2017 objective) decodes raw hyperbolic distances and minimises a soft-ranking NLL; its negatives are **re-sampled every step** inside the loss (`_loss_from_dist` тЖТ `ranking_nll_from_dist`; `sample_negatives` draws `n_negatives` per node from the *known* non-edges `A_ik==0`, excluding self and imputed unknown pairs), so the loss is stochastic тАФ tests compare smoothed loss windows, not `hist[-1] < hist[0]`. Each positive pair is weighted by `A_ij`, which is `1` for ordinary edges and the imputed value for unknown edges, so gradients reach `a_Omega` through the joint optimiser. `"fermi_dirac"` instead decodes the Fermi-Dirac edge probability `1/(exp((d_HтИТr)/t)+1)` and minimises Bernoulli cross-entropy (the same family as HyperMap/D-Mercator but with a single global `r`, `t`); only this mode reports `is_generative()=True`. `decode()` returns the distance matrix under `"ranking"` and the probability matrix under `"fermi_dirac"`.
- **Lorentz Embeddings = the Poincar├й-Embeddings ranking objective on the hyperboloid (Nickel & Kiela 2018)**: same soft-ranking loss, different (numerically better) chart of the same space, with an exact-geodesic optimiser. `LorentzEmbeddingsEmbedder` embeds on `manifolds.lorentz.LORENTZ` (`StableLorentz`, the spatial-norm-clamped `geoopt.Lorentz`, whose `expmap` is the paper's Eq. 9), so intrinsic dim `d` uses `d+1` ambient coords. Near-origin init is Eq. 6 (`x' ~ U(-init_scale, init_scale)`, `x_0 = sqrt(1+тАЦx'тАЦ┬▓)`, `init_scale=1e-3` per the paper). `embeddings()` returns the **Poincar├й-ball image** `(N,d)` via the Eq. 11 isometry (`lorentz_to_poincare`) so it drops into the disk plotters and the rest of the library; raw `(N,d+1)` hyperboloid coords are on `hyperboloid_embeddings()`, and `X_init`/warm-start (`update()`) use the hyperboloid representation, not the ball. **Similarity generalisation**: `structural_similarity` is the *weighted* adjacency `K`, and the negative set is the paper's `N(i,j)={z: K_{iz}<K_{ij}}` (less-similar nodes), which collapses to Poincar├й Embeddings' non-edges on a binary graph тАФ this is how weighted graphs are supported. **Unknown edges (the key design call)**: the `K_{iz}<K_{ij}` membership test is a non-differentiable hard threshold (same obstruction that blocks Hydra's shortest-path target from routing unknown-edge gradients), so (1) unknown pairs are excluded from every `N(i,j)` тАФ negatives come only from *known* structure, keeping `a_Omega` out of the combinatorial part тАФ and (2) positives are weighted by `K_{ij}` **only when `Omega` is non-empty** (`_loss_from_dist` sets `weighted = len(unknown_edges) > 0`), which routes a smooth gradient to `a_Omega`. Consequence: with `Omega={}` the loss is the paper's *unweighted* Eq. 12 exactly (binary *and* weighted graphs); the `K_{ij}`-weighting is hypeGRL's device confined to the unknown-edge regime the paper never covers (and a no-op on binary graphs). **Difference from the paper**: optimiser is geoopt `RiemannianAdam`, not the paper's exact-geodesic Riemannian *SGD* (Algorithm 1) тАФ geometry still exact, update rule Adam, matching the rest of the library. Ranking-only (the paper's sole loss); no Fermi-Dirac variant, so `is_generative()=False` and `decode()` returns the Lorentz distance matrix. **Verified against the theSage21 PyTorch reference** (`scripts/lorentz-embeddings`): its `N(i,j)` (`pairwise[i,x] < min`) and unweighted-positive loss confirm our similarity-generalisation and `Omega`-gated weighting; it treats the matrix as a *directed* DAG and samples one positive per anchor per epoch (minibatch SGD), where we sum symmetrically over all edges full-batch (closer to the paper's symmetric Eq. 12). **Defaults `lr_X=0.3` / `n_steps=2000` are tuned, not paper values**: leaves must reach large radius for generality=norm to emerge, and a timid rate leaves the embedding under-spread (~5-10 AUC points worse on trees/karate/Les Mis). This aggressive rate is only safe because of the `StableLorentz` clamp (`max_norm`, a constructor arg; the embedder builds its own manifold instance, default `1e3` тАФ sweep-tuned, see the manifolds note) тАФ without it, `lr_XтЙ│0.5` NaNs (the reference guards the same way via `set_dim0`). We deliberately skip the reference's lr burn-in (no measurable effect under Adam).
